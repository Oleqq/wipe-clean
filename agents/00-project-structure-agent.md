# Агент: Project Structure Map

## Роль

Фиксирует реальную структуру проекта `a4-remont` для LLM-разработки.

Задача этого агента:

- быстро ввести модель в архитектуру текущей темы;
- показать, как сюда переносить новые дизайн-макеты;
- зафиксировать правильные точки входа для разработки статики, секций и интеграции в WordPress;
- не дать писать код в "похожее место", когда у проекта уже есть устоявшаяся структура.

## Корень проекта

- `wp-content/themes/a4-remont/` — основная кастомная тема.
- `agents/` — агентские markdown-доки для LLM.
- `SEO-Audit/`, `SEO/` — сопутствующие материалы по аудиту и SEO.
- `scripts/` — локальные утилиты сборки и минификации.

## Точки входа темы

- `functions.php` — bootstrap темы; здесь подключаются все модули из `inc/`.
- `header.php`, `footer.php` — общие обертки фронта.
- `front-page.php`, `single-*.php`, `archive-*.php`, `page.php`, `index.php` — основные шаблоны WordPress-маршрутов.
- `page-templates/*.php` — шаблоны отдельных обычных страниц.

## Модульная зона `inc/`

### Базовые системные модули

- `setup.php` — базовая инициализация темы.
- `slugs.php` — контроль slug и служебных URL.
- `media.php` — правила работы с медиа, `alt`, `title`, image helpers.
- `performance.php` — легкие performance-оптимизации темы.
- `enqueue.php` — схема подключения CSS/JS, зависимостей секций и выбор `.min`-файлов.
- `content-types.php` — регистрация CPT/таксономий и SEO-safe логика архивов.
- `section-helpers.php` — общие helper-функции для секций.
- `static-content.php` — fallback-контент и fallback-ветки.

### UI и данные проекта

- `header-builder.php`, `footer-builder.php` — сборка шапки и подвала.
- `popups.php` — рендер и логика модальных окон.
- `cookie-notice.php` — cookie-баннер.
- `form-submissions.php` — админ-панель заявок и интеграция заявок из CF7.
- `admin-docs.php` — встроенная база знаний/инструкции в админке.

### SEO

- `seo-keywords.php` — keywords-поля для редакторов и архивов.
- `acf-seo-social.php` — ACF option page для глобальных OG-настроек.
- `seo-social.php` — fallback/override Open Graph и Twitter meta.

### ACF-слой

- `acf.php` — общие ACF helper-структуры.
- `acf-*.php` — локальные PHP-описания групп полей по сущностям:
  - страницы;
  - архивы;
  - single CPT;
  - глобальные option pages.

Правило: основной источник ACF в проекте — код в `inc/acf-*.php`, а не ручная настройка через GUI.

## Шаблоны секций

- `template-parts/section/` — рабочие PHP-секции проекта.
- `template-parts/static/section/` — статические fallback-версии секций.
- `template-parts/components/` — переиспользуемые части, если они есть.
- `template-parts/global/` — общие фрагменты, если они есть.

Каждая секция должна мыслиться как связка:

1. PHP-шаблон секции.
2. ACF layout/данные.
3. CSS chunk.
4. JS-модуль при необходимости.
5. fallback-разметка.

## CSS-архитектура

- `css/base.css` — базовый общий слой.
- `css/section/*.css` — посекционные стили.
- `css/vendor/` — vendor CSS, не минифицируется проектной логикой.
- `css/style.css` — дополнительный глобальный слой темы.
- `style.css` в корне темы — финальный stylesheet WordPress-темы.

Текущая модель подключения:

1. fonts/vendor;
2. `css/base.css`;
3. `css/section/*.css` по набору секций страницы;
4. `style.css` как override-слой.

Контроллер этого поведения — `inc/enqueue.php`.

## JS-архитектура

- `js/script.js` — единая точка входа фронта.
- `js/modules/` — глобальные модули сайта.
- `js/section/` — JS по конкретным секциям.
- `js/vendor/` — vendor JS.

Правило:

- секционный JS добавляется только если у секции реально есть интерактив;
- не переносить шаблонные HTML-решения в общий JS без необходимости;
- если поведение можно сделать в шаблоне, делать его в шаблоне.

## Минификация и ассеты

- рядом с основными theme assets лежат `.min.css` и `.min.js`;
- тема предпочитает `.min`-файл, если он существует;
- vendor-файлы пропускаются и не минифицируются проектной логикой;
- W3 Total Cache minify не должен быть главным источником минификации темы.

## Как интегрируется новый дизайн-макет в WordPress

Нормальный путь интеграции нового макета или нового блока:

1. взять статический блок;
2. перенести его в `template-parts/section/<section>.php`;
3. вынести fallback в `template-parts/static/section/...`;
4. описать поля в `inc/acf-*.php`;
5. подключить секцию в нужный builder, archive, page-template или single;
6. добавить CSS chunk в `css/section/`;
7. зарегистрировать зависимости стилей в `inc/enqueue.php`, если секция использует чужие классы;
8. добавить JS только при необходимости.

Ненормальный путь:

- править только статику и ожидать, что это станет рабочей секцией темы;
- складывать новую бизнес-логику в `style.css`, минуя секционный слой;
- дублировать секции вместо расширения существующих.

## Важные маршруты контента

- обычные страницы — `page-templates/*.php` и/или `page.php`;
- услуги — `single-service.php`, `archive-service.php`;
- новости — `single-news.php`, `archive-news.php`;
- работы — `single-work.php`, `archive-work.php`;
- отзывы — `archive-feedback.php` и секционные выводы отзывов;
- глобальные экраны — option pages через ACF.

## Что должен понять LLM перед началом работы

1. Это не "любой WordPress-проект", а тема с жесткой секционной архитектурой.
2. Новые макеты нужно встраивать в существующую систему, а не строить рядом вторую.
3. Основной фокус — переиспользование текущих паттернов темы.
4. Любое решение должно быть безопасным для последующей заливки темы на прод без потери контентных данных.

## Жесткие правила

- Не ломать контентные данные на проде: ACF values, alt, title, тексты, slug, Yoast-поля.
- Не делать ACF GUI главным источником правды, если группа уже описана в `inc/acf-*.php`.
- Не считать статику из `template-parts/static` основной рабочей веткой.
- Не вносить секционные зависимости стилей скрыто: их нужно фиксировать в `inc/enqueue.php`.
- Не плодить обходной JS, если проблема решается в шаблоне или PHP-слое.
