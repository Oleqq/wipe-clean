# Агент: Project Structure Map

## Роль

Фиксирует реальную структуру проекта `a4-remont` для LLM-разработки.

Главная задача этого агента: не просто описать дерево файлов, а зафиксировать **двухэтапную модель проекта**:

1. сначала создается **полный статический слой** в `/static`;
2. только потом эта статика переносится в WordPress-тему.

Если агент этого не понял, он почти неизбежно начнет смешивать верстку и интеграцию, а это для проекта неверно.

## Главная архитектурная модель проекта

Проект состоит из двух разных контуров разработки.

### Контур 1. Полная статика

Это первый и обязательный этап.

Источник:

- `static/`

Задача этого слоя:

- собрать новый сайт или новый макет целиком в статике;
- выстроить HTML-структуру страниц;
- собрать секции;
- написать SCSS;
- реализовать первичный JS;
- подготовить чистый, автономный фронтовый слой без WordPress-зависимостей.

### Контур 2. WordPress-тема

Это второй этап.

Источник:

- `wp-content/themes/a4-remont/`

Задача этого слоя:

- взять уже готовую статику;
- перенести ее в архитектуру темы;
- подключить данные через ACF;
- встроить секции в archive/single/page-template;
- сохранить fallback и текущую логику проекта.

Ключевое правило:  
**тема не является местом первичной черновой верстки.**

## Корень проекта

- `static/` — полный статический слой проекта.
- `wp-content/themes/a4-remont/` — рабочая WordPress-тема.
- `agents/` — агентские markdown-доки для LLM.
- `SEO-Audit/`, `SEO/` — сопутствующие материалы по аудиту и SEO.
- `scripts/` — локальные утилиты сборки и минификации.

## Как устроен статический слой `/static`

Это не вспомогательная папка, а полноценный первый источник нового макета.

### Основные каталоги

- `static/template-parts/pug/` — исходные Pug-шаблоны.
- `static/template-parts/html/` — статические HTML-версии страниц и секций.
- `static/scss/` — SCSS проекта.
- `static/js/` — JS для статической сборки.
- `static/images/`, `static/webp-images/`, `static/fonts/` — ассеты.

### Что уже видно по структуре

В `/static` есть:

- страницы:
  - `index.html`
  - `about-us.html`
  - `contacts.html`
  - `news.html`
  - `news-single.html`
  - `services.html`
  - `service-single.html`
  - `our-works.html`
  - `work-single.html`
  - `reviews.html`
  - `faq.html`
  - `404.html`
  - `privacy-policy.html`
  - `payment-delivery.html`
- Pug-страницы: `static/template-parts/pug/pages/*.pug`
- Pug-секции: `static/template-parts/pug/section/**/*.pug`
- HTML-секции: `static/template-parts/html/section/**/*.html`
- базовые SCSS-файлы:
  - `static/scss/style.scss`
  - `static/scss/base-only.scss`

### Что должен понимать агент про `/static`

1. Новый макет сначала оформляется здесь.
2. Здесь можно свободно собирать страницу как полноценный статический продукт.
3. Здесь нельзя тащить WordPress-специфику раньше времени.
4. Имена блоков, структура секций и паттерны классов отсюда потом должны переноситься в тему.

## Как устроена WordPress-тема

### Точки входа темы

- `functions.php` — bootstrap темы; здесь подключаются все модули из `inc/`.
- `header.php`, `footer.php` — общие обертки фронта.
- `front-page.php`, `single-*.php`, `archive-*.php`, `page.php`, `index.php` — основные шаблоны WordPress-маршрутов.
- `page-templates/*.php` — шаблоны отдельных обычных страниц.

### Модульная зона `inc/`

#### Базовые системные модули

- `setup.php` — базовая инициализация темы.
- `slugs.php` — контроль slug и служебных URL.
- `media.php` — правила работы с медиа, `alt`, `title`, image helpers.
- `performance.php` — легкие performance-оптимизации темы.
- `enqueue.php` — схема подключения CSS/JS, зависимостей секций и выбор `.min`-файлов.
- `content-types.php` — регистрация CPT/таксономий и SEO-safe логика архивов.
- `section-helpers.php` — общие helper-функции для секций.
- `static-content.php` — fallback-контент и fallback-ветки.

#### UI и проектные модули

- `header-builder.php`, `footer-builder.php` — сборка шапки и подвала.
- `popups.php` — модальные окна.
- `cookie-notice.php` — cookie-баннер.
- `form-submissions.php` — панель заявок и интеграция с CF7.
- `admin-docs.php` — встроенная база знаний в админке.

#### SEO

- `seo-keywords.php` — keywords-поля для редакторов и архивов.
- `acf-seo-social.php` — option page для OG-настроек.
- `seo-social.php` — fallback/override Open Graph и Twitter meta.

#### ACF-слой

- `acf.php` — общие ACF helper-структуры.
- `acf-*.php` — локальные PHP-описания групп полей по сущностям:
  - страницы;
  - архивы;
  - single CPT;
  - глобальные option pages.

Правило: основной источник ACF в проекте — код в `inc/acf-*.php`, а не ручная настройка через GUI.

## Шаблоны секций темы

- `template-parts/section/` — рабочие PHP-секции проекта.
- `template-parts/static/section/` — static fallback внутри темы.
- `template-parts/components/` — переиспользуемые части, если они есть.
- `template-parts/global/` — общие фрагменты, если они есть.

Каждая секция в теме должна мыслиться как связка:

1. PHP-шаблон секции.
2. ACF layout/данные.
3. CSS chunk.
4. JS-модуль при необходимости.
5. fallback-разметка.

## CSS-архитектура темы

- `css/base.css` — базовый общий слой.
- `css/section/*.css` — посекционные стили.
- `css/vendor/` — vendor CSS.
- `css/style.css` — дополнительный глобальный слой темы.
- `style.css` в корне темы — финальный stylesheet WordPress-темы.

### Реальная модель подключения

1. fonts/vendor;
2. `css/base.css`;
3. `css/section/*.css` по набору секций страницы;
4. `style.css` как override-слой.

Контроллер этого поведения — `inc/enqueue.php`.

## JS-архитектура темы

- `js/script.js` — единая точка входа фронта.
- `js/modules/` — глобальные модули сайта.
- `js/section/` — JS по конкретным секциям.
- `js/vendor/` — vendor JS.

Правило:

- секционный JS добавляется только если у секции реально есть интерактив;
- если проблему можно решить в шаблоне или PHP-слое, не надо плодить обходной JS.

## Минификация и ассеты

- рядом с основными theme assets лежат `.min.css` и `.min.js`;
- тема предпочитает `.min`-файл, если он существует;
- vendor-файлы проектной минификацией не обрабатываются;
- W3 Total Cache minify не должен быть главным источником минификации темы.

## Правильный маршрут разработки нового макета

### Этап 1. Верстка

1. Макет собирается в `/static`.
2. Страница формируется как статическая страница.
3. Секции формируются как статические блоки.
4. Имена блоков и структура секций стабилизируются.
5. JS и адаптив доводятся на статическом слое.

### Этап 2. Интеграция

1. Берется уже готовая статика.
2. Она переносится в `template-parts/section/<section>.php`.
3. Fallback переносится в `template-parts/static/section/...`.
4. Поля описываются в `inc/acf-*.php`.
5. Секция подключается в нужный builder, archive, page-template или single.
6. CSS переносится в `css/section/`.
7. Зависимости стилей фиксируются в `inc/enqueue.php`, если секция использует чужие классы.
8. JS переносится в тему только если он реально нужен на этапе интеграции.

## Что считается неправильным маршрутом

- начинать WordPress-интеграцию до того, как собран static-слой;
- делать "черновую верстку" сразу в PHP-шаблоне темы;
- править только `template-parts/static` и считать, что секция интегрирована;
- складывать новую бизнес-логику в `style.css`, минуя посекционный слой;
- дублировать секции вместо расширения существующих.

## Важные маршруты контента

- обычные страницы — `page-templates/*.php` и/или `page.php`;
- услуги — `single-service.php`, `archive-service.php`;
- новости — `single-news.php`, `archive-news.php`;
- работы — `single-work.php`, `archive-work.php`;
- отзывы — `archive-feedback.php` и секционные выводы отзывов;
- глобальные экраны — option pages через ACF.

## Что должен понять LLM перед началом работы

1. Проект сознательно разделяет **верстку** и **интеграцию**.
2. `/static` — это не мусорная папка и не legacy-слой, а первый обязательный этап.
3. `wp-content/themes/a4-remont` — это слой переноса уже готовой статики в WordPress.
4. Основной фокус — переиспользование проектных паттернов, а не придумывание новых структур под каждый макет.
5. Любое решение должно быть безопасным для последующей заливки темы на прод без потери контентных данных.

## Жесткие правила

- Не ломать контентные данные на проде: ACF values, alt, title, тексты, slug, Yoast-поля.
- Не делать ACF GUI главным источником правды, если группа уже описана в `inc/acf-*.php`.
- Не считать `template-parts/static` основным рабочим слоем интеграции.
- Не вносить секционные зависимости стилей скрыто: их нужно фиксировать в `inc/enqueue.php`.
- Не смешивать этап верстки и этап интеграции в одном рабочем решении.
